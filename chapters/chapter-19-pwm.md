### 19.1 回顾：你在 18.9 节做了什么

上一章结尾的小实验里，你用手动循环 `led.value(1)` `sleep(1ms)` `led.value(0)` `sleep(9ms)` 的方式，让 LED "看起来"只有 10% 亮度。

这个方法能work，但有三个问题：

1. **占 CPU**：`while True` 一直在跑，ESP32 没法干别的事情
2. **频率不稳定**：`sleep()` 的精度有限，代码一复杂就抖动
3. **太累人**：想调亮度还得改代码里的数字

ESP32 芯片上有一个专门的**硬件 PWM 模块**，能帮你自动完成"快速开关"这件事——你只需要告诉它**频率**和**占空比**，它自己在后台闪，不需要 `while True`、不需要 `sleep()`。

### 19.2 占空比和频率——快速回顾

从 18.9 节你已经理解了这两个概念：

- **占空比** = 亮的时间 ÷ 总周期。10% 是暗的，50% 是半亮，90% 是很亮
- **频率** = 1 ÷ 总周期。100Hz 已经超过人眼临界值，看不到闪烁

硬件 PWM 把这两个参数明确为函数调用：

```python
from machine import Pin, PWM

led = PWM(Pin(13), freq=5000)    # 频率设为 5000Hz
led.duty(512)                    # 占空比设为 50%
```

- `freq=5000`：频率 5000Hz，比你手动写的 100Hz 快 50 倍——亮度更平滑
- `duty(512)`：占空比，取值范围 0~1023（为什么是 1023 而不是 100？见下一节）

**对比你手动实现的版本：**

| 方式 | 频率 | CPU | 精度 |
|------|------|-----|------|
| 18.9 手动 `sleep()` | ~100Hz | 100% 占用 | 粗略 |
| 19章硬件 PWM | 5000Hz | 几乎不占 | 1024 级 |

**关于 `duty` 为什么是 0~1023 而不是 0~100**

ESP32 的 PWM 有 **10 位分辨率**——意思是把高电平时间分成 2¹⁰ = 1024 个刻度。0 = 0%、512 = 50%、1023 = 100%。

> 你可以当成百分比来理解——`duty(0)` = 0%，`duty(512)` ≈ 50%，`duty(1023)` = 100%。

### 19.3 呼吸灯——用硬件 PWM 实现渐变

呼吸灯的原理很简单：**让占空比从 0 慢慢涨到 1023（渐亮），再从 1023 慢慢降到 0（渐暗），不断循环。**

```python
from machine import Pin, PWM
import time

led = PWM(Pin(13), freq=5000)

while True:
    for duty in range(0, 1024, 4):    # 0 → 1023，每次加 4
        led.duty(duty)
        time.sleep(0.005)
    for duty in range(1023, -1, -4):  # 1023 → 0，每次减 4
        led.duty(duty)
        time.sleep(0.005)
```

**逐行理解：**

- `PWM(Pin(13), freq=5000)`：将 GPIO13 设为 PWM 输出，频率 5000Hz
- `range(0, 1024, 4)`：从 0 到 1023，步长为 4——总共 256 个亮度台阶
- `led.duty(duty)`：设置 PWM 占空比为当前值
- `time.sleep(0.005)`：每个台阶停留 5 毫秒——换得平滑的渐变感
- 整个渐变周期约为 `256 × 2 × 5ms = 2.56 秒`

> 试试改步长：把 4 改成 8 → 速度快一倍，但台阶感更强。把 4 改成 1 → 最平滑，但周期变长。

💡 **思考延续**

PWM 是你已经每天都在用的技术。调节手机屏幕亮度时，屏幕背光 LED 并没有真正变暗——它只是以几千赫兹的频率在"亮"和"灭"之间快速切换。你觉得是"50% 亮度"，实际上是 50% 亮 + 50% 灭——你的眼睛来不及分辨，自动帮它们"平均"了。

同样的原理也用于：笔记本风扇调速、电饭煲温控、电动自行车的加减速。PWM 之所以普遍，是因为它只用开/关一个引脚（不需要真正的模拟信号），却能精确模拟"不多不少刚刚好"的效果。
